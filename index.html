홈 화면 쿼리 튜닝 - 비정규화를 통한 레이턴시 감소

요구 사항

홈 화면에서 상점 정보와 함께 가장 저렴한 코스를 조회하는 서브쿼리의 성능 저하를 인지하고, 쿼리 최적화가 필요함.

해결 방안

  • 가장 저렴한 코스 가격을 미리 계산하여 store 테이블에 비정규화하여 저장
  • 상점 이미지(URL)를 JSON 배열 형태로 store 테이블에 저장, 별도 테이블 조회 없이 즉시 활용 가능

성과

  • 레이턴시 50% 이상 감소 → 기존 200ms 후반에서 100ms 중~후반으로 단축
  • 서브쿼리 및 조인 쿼리 호출을 줄여 데이터 조회 성능 개선 및 데이터베이스 부하 감소

회원 탈퇴 API 쿼리 성능 최적화 - ORM N+1 쿼리 튜닝

요구 사항

회원 탈퇴 API 구현 중, 관계형 테이블의 깊이가 증가함에 따라 비정상적인 슬로우 쿼리 발생.

데이터 삭제 시 N+1 문제로 인해 불필요한 추가 조회가 발생하여 응답 속도가 급격히 저하되는 문제를 해결해야 함.

해결 방안

  • 쿼리 로깅 분석 → 회원 탈퇴 API 호출 시 TypeORM이 실행하는 SQL 로그를 수집하여 병목 지점 파악
  • N+1 문제 인지 → 연관 관계의 데이터가 개별적으로 조회되면서 불필요한 추가 쿼리 발생 확인
  • Raw Query 최적화 → TypeORM의 QueryBuilder를 활용하여 단일 쿼리로 일괄 삭제 처리, 필요 없는 조회 쿼리 제거

성과

  • 쿼리 응답 속도 99% 개선 → 최대 120,000ms → 100ms 미만으로 레이턴시 감소
  • 데이터베이스 부하 감소 → 불필요한 조회 쿼리 제거로 트랜잭션 처리 성능 향상 및 리소스 절약
  • UX 개선 → 회원 탈퇴 API의 응답 속도가 획기적으로 단축됨에 따라 사용자 경험(UX) 향상

CI/CD - 도메인별 독립적인 배포 전략

요구 사항

세 개의 독립적인 도메인(customer, partner, admin)을 분리된 서버에서 개별적으로 관리하면서,

레퍼런스가 적은 NestJS Monorepo 환경에서 CI/CD 프로세스를 확립해야 함.

해결 방안

  • 파일 변화 감지 → 각 도메인별 폴더에서 파일 변화를 감지하여, 해당 변화에 맞는 워크플로우를 실행하는 방식으로 server monorepo의 문제를 해결
  • 컨테이너 기반 배포 → AWS ECS(Elastic Container Service) 및 ECR(Elastic Container Registry)을 활용하여 각 도메인을 독립적인 Task로 배포
  • 격리된 서비스 운영 → 특정 도메인에서 장애가 발생하더라도 다른 도메인에 영향을 미치지 않는 구조로 안정성 확보
  • AWS Load Balancer 활용 → 요청을 받는 도메인에 따라 트래픽을 자동 분기하여 각각의 ECS 서비스로 전달, 확장성과 유연성 확보

성과

  • 최적화된 도커 이미지 관리 → 공통 의존성을 모듈화하고 도메인별로 컨테이너를 분리하여 도커 이미지 용량 70% 감소 (560MB ⇒ 118.81MB)
  • 빌드 속도 개선 → Docker BuildX 패키지를 활용한 이미지 빌드 캐싱을 적용하여 배포 시간 50% 이상 단축 (13분 ⇒ 6분 30초~7분)
  • 트래픽 라우팅 최적화 → AWS Load Balancer를 활용하여 도메인별 요청을 자동으로 적절한 컨테이너로 분배, 서비스 가용성 및 확장성 강화

NestJS Monorepo Architecture - 분리된 니즈에 알맞은 아키텍처 선정

요구 사항

고객(customer), 파트너(partner), 관리자(admin) 등 세 개의 독립적인 도메인을 단일 코드베이스에서 효율적으로 관리하면서, 공통 자산을 효과적으로 공유하고 유지보수성을 극대화할 수 있는 구조가 필요


해결 방안

  • 모노레포 아키텍처 도입 → 단일 저장소에서 여러 도메인을 관리하면서, 공통 자산(Entity, Interface, Util 함수 등)을 모듈화하여 재사용 가능
  • 직관적인 의존성 주입 → 도메인별 필요한 모듈만을 가져올 수 있도록 구성하여, 코드 간 결합도를 낮추고 유지보수성을 향상
  • 중복 코드 감소 및 일관성 유지 → 서비스 간 중복 코드를 최소화하고, 공통 기능을 중앙에서 관리하여 개발 생산성 향상

꾹 페이 리팩토링 - 확장성과 가독성을 모두 잡은 코드 리팩토링

요구 사항

결제 시스템이 단순 결제를 넘어 검증, 수수료 계산 등 다양한 비즈니스 로직을 포함하면서 코드량이 급격히 증가.

이로 인해 가독성이 저하되고, 새로운 기능 추가 시 유지보수가 어려운 문제가 발생함.

해결 방안

  • Template Method 패턴 활용 → 결제 프로세스를 추상화해 공통 로직을 재사용하고 유지보수성 강화
  • Got 라이브러리 활용 → 기존 Timeout의 종류를 나누지 않은 코드에서 Read Timeout과 Connection Timeout을 분리하여 분기 처리

성과

  • 불필요한 IF문 제거 → 조건 분기를 함수로 캡슐화하여 가독성을 개선하고 코드 라인을 50줄 이상 감소
  • 확장성 증가 → 새로운 결제 방식 추가 시, 추상 클래스만 상속받아 구현 가능
  • Timeout 처리 최적화 → Read Timeout 시 사내 Slack을 통해 결제 상태를 실시간 모니터링하고, Connection Timeout은 2초 미만의 랜덤 재시도를 3회 진행하여 클라이언트 편의성 향상
  • 휴먼 에러 100% 감소 → 정형화된 추상 클래스를 기반으로 구현하여 실수 방지
  • 결제 장애 0% → 현재 2025.03.29까지 발견된 결제 장애 0건

완벽히 사람과 동일한 품질의 콘텐츠를 생성하는 것은 어렵지만, 자동화된 포스팅 매크로를 도입하여

자동 원고 발행 - OpenAI(GPT)를 활용한 콘텐츠 자동화

요구 사항

마케팅 팀이 수작업으로 원고를 작성해야 한다면 자동화의 의미가 없음을 인지하고, GPT를 활용한 자동 원고 발행 시스템을 구축하여 업무 효율성을 높이고자 함.

해결 방안

  • GPT-4o를 활용하여 특정 키워드를 기반으로 콘텐츠(본문) 자동 생성
  • GPT-4 Turbo를 활용하여 SEO 최적화된 블로그 타이틀 50가지 자동 생성

성과

  • 마케팅 팀의 블로그 발행 업무량 50% 이상 감소 (기존 하루 10~15건 → 5건)
  • SEO 최적화된 키워드(타이틀) 자동 선정으로 키워드 탐색 시간 단축
  • 블로그 상위 노출 유지 문제 해결 과제 도출 (1페이지 상단 5위 안 유지 필요)
